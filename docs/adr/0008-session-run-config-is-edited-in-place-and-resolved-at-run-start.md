# Session Run Config Is Edited In Place And Resolved At Run Start

Session run config is a single durable row that surfaces edit in place, and runs resolve their effective config by requiring that row and parsing the schema-defaulted typed row when generation starts. Sparse stored cells recover through the session config table defaults; a missing config row is still a broken session invariant. Typed per-cell writes to the session config row are part of the RunConfigs interface — a validated cell write is a config edit, and web surfaces bind to cells directly rather than going through mediated setters. There is no per-run override layer: `runs.generate` takes no config argument, and CLI flags like `-M` intentionally persist onto the session config rather than acting as one-off overrides, with run records snapshotting the resolved config for auditability. RunConfigs owns every operation a single cell write cannot express — creation merge, structured updates, the new-session default, prompt unlinking, and run-start resolution. We chose this because TinyBase's reactive cell binding already gives validated two-way writes, mediating them would add a layer with no invariant to protect, and the unused override layer existed only speculatively while duplicated merge logic was the real cost.
